# Provenance Guard

A backend system that classifies whether submitted text-based creative work was
written by a human or generated by AI, scores confidence in that classification,
surfaces a plain-language transparency label, and lets creators appeal — with
rate limiting and a structured audit log.

The full architecture diagram and the five spec questions live in
[`planning.md`](planning.md).

## Architecture Overview

The path a single submission takes from input to the transparency label a reader
sees:

```
POST /submit {text, creator_id}
   │ mint content_id (UUID)
   │ raw text
   ├─► Signal 1: LLM-as-judge (Groq)  ─► P_ai (semantic)  ─┐
   ├─► Signal 2: stylometric (Python) ─► P_ai (structural) ─┤
   │                                                        ▼
   │                    combine  confidence = 0.6·P_llm + 0.4·P_stylo
   │                                                        │
   │                    threshold ─► attribution (likely_ai / uncertain / likely_human)
   │                                                        │
   │                    transparency label (reader-facing text with %)
   │                                                        │
   │   audit log ◄── content_id, confidence, both signal scores, status ──┘
   ▼
response {content_id, attribution, confidence, label, signal_scores}
```

1. A creator POSTs text to `/submit`; the API mints a `content_id`.
2. The raw text fans out to **two independent detectors** — the Groq LLM judge
   (semantic read) and the stylometric heuristics (structural stats) — each
   returning an estimated probability the text is AI-generated.
3. The two scores are combined (0.6 LLM / 0.4 stylometric) into a single
   **confidence**, which a threshold maps to an **attribution** category and a
   plain-language **transparency label**.
4. The decision (confidence + both signal scores + status) is written to the
   **audit log**, and the JSON response returns to the caller.
5. If the creator disputes the result, `/appeal` sets the status to
   `under_review` and logs their reasoning alongside the original decision.

API directly with `curl`.

## Web UI

A single self-contained page (`static/index.html`, served at `/`, no extra
dependencies) with three tabs:

- **Analyze** — paste text, see the color-coded transparency label, a confidence
  meter, and the two individual signal scores (LLM vs. stylometric).
- **Appeal** — the `content_id` from an analysis is prefilled; enter reasoning to
  file an appeal.
- **Audit Log** — a live table of recent decisions and appeals.

## Detection Signals

The pipeline uses **two genuinely independent signals** — one semantic, one
structural. That independence is the point: they capture different properties of
the text, so their agreement is informative and their disagreement is itself an
honest uncertainty signal.

**Signal 1 — LLM-as-judge (Groq `llama-3.3-70b-versatile`).** The text is sent to
the model with a prompt asking for a calibrated probability that it was
AI-generated, returned as JSON. It captures *holistic* cues a human reader
notices — smoothed register, boilerplate hedging ("it is important to note"),
over-even argument structure, absence of personal voice. **What it misses:** it
is a notoriously unreliable detector — it is biased toward flagging formal or
non-native-English human writing as AI, and it is fooled by lightly-edited AI
text. The prompt explicitly instructs the model *not* to raise its score for mere
formality, but the bias cannot be fully removed.

**Signal 2 — Stylometric heuristics (pure Python, no API).** Computes structural
statistics: sentence-length variance (burstiness), type-token ratio (vocabulary
diversity), and punctuation density. Human writing is *bursty* — it mixes short
and long sentences; AI writing trends *uniform*. **What it misses:** it
understands zero meaning, its statistics are meaningless on short texts, and
genre confounds it — a listicle, a formal essay, and a repetition-heavy poem all
look "uniform" regardless of who wrote them.

**Why this pairing:** the LLM catches AI that is structurally varied but
semantically generic (the academic paragraph in the scoring examples); the
stylometric signal is a cheap, API-free, deterministic cross-check that can pull
an over-confident LLM back toward uncertainty. Neither is trustworthy alone; the
combination, plus a deliberately wide "uncertain" band, is.

**If deploying for real,** I would: (1) replace the fixed 0.6/0.4 weights with
weights calibrated against a labeled dataset; (2) add a perplexity-based signal
(a third, independent statistical view); and (3) never surface a hard "AI"
verdict without the appeal path, because the false-positive cost to a wrongly
accused human writer is severe.

## Confidence Scoring

`confidence` = P(text is AI-generated) = `0.6 · P_llm + 0.4 · P_stylometric`,
mapped to a category by threshold: `>= 0.75` → likely AI, `<= 0.40` → likely
human, in between → uncertain. The LLM (semantic) signal leads; the stylometric
(structural) signal is an independent cross-check. Short samples are damped
toward 0.5 because their statistics are unreliable.

**How I validated it's meaningful.** I tested five deliberately chosen inputs
spanning the range — clearly-AI, clearly-human, and borderline cases (formal
human, edited AI) — and checked three things: (1) the ordering matched intuition
(clearly-AI scored well above clearly-human, a 0.48 gap); (2) all three label
categories were actually reachable, not just two around a 0.5 flip; and (3) I
printed **both signal scores separately** for each input so a misbehaving signal
would be visible rather than hidden inside the combined number. That last check is
what surfaced the honest-disagreement cases below.

Two submissions with noticeably different confidence (from that testing):

| Submission | LLM score | Stylometric score | Confidence | Result |
| --- | --- | --- | --- | --- |
| Uniform marketing copy ("Our platform delivers powerful results…") | 0.90 | 0.68 | **0.81** | Likely AI-generated |
| Casual restaurant review ("ok so i finally tried that ramen place…") | 0.10 | 0.17 | **0.13** | Likely human-written |

The scoring also surfaces honest disagreement: a polished academic paragraph read
as AI *semantically* (LLM 0.80) but *structurally* human (stylometrics 0.34, it
was sentence-length-varied), combining to 0.61 → **Uncertain** rather than a false
verdict. A formal (non-native-style) human paragraph likewise landed at 0.55 →
**Uncertain**, not accused — the wide uncertain band absorbing the false-positive
risk by design.

## Transparency Label Variants

The label a reader sees. `{ai}` = `round(confidence * 100)`, `{human}` = `100 - {ai}`,
where `confidence` is the estimated probability the text is AI-generated.

| Variant | Shown when | Exact text |
| --- | --- | --- |
| **High-confidence AI** | `confidence >= 0.75` | `⚠️ Likely AI-generated ({ai}% confidence). This text shows strong signals of AI authorship.` |
| **High-confidence human** | `confidence <= 0.40` | `✅ Likely human-written ({human}% confidence). No strong signals of AI authorship were found.` |
| **Uncertain** | `0.40 < confidence < 0.75` | `❓ Attribution uncertain ({ai}% estimated AI). Our signals are mixed, so this result is inconclusive.` |

The AI variant says "shows strong signals of" rather than accusing, reflecting
the design choice that a false positive (labeling human work as AI) is the
costlier error.

## Appeals Workflow

A creator who believes a classification is wrong POSTs their `content_id` and
free-text `creator_reasoning` to `/appeal`. The system:

1. Looks up the original decision for that `content_id` (`find_latest`).
2. Appends an appeal record to the audit log with `status: "under_review"`, the
   `appeal_reasoning`, and the linked `original_attribution` / `original_confidence`.
3. Returns a confirmation. **No automated re-classification** — a human reviewer
   works the queue by filtering `GET /log` for `status == "under_review"`, seeing
   the creator's reasoning next to the original signal scores.

```bash
curl -s -X POST http://localhost:5000/appeal \
  -H "Content-Type: application/json" \
  -d '{"content_id": "609837be-...", "creator_reasoning": "I wrote this myself; English is my second language so my style is plain."}'
# -> {"content_id": "609837be-...", "status": "under_review",
#     "message": "Your appeal was received and is under review."}
```

## Rate Limiting

`POST /submit` is rate limited per client IP (Flask-Limiter, in-memory store):

```python
@limiter.limit("10 per minute;100 per day")
```

**Reasoning for these values:**
- **10 per minute (burst guard):** No legitimate creator submits ten pieces of
  writing in a minute — submission is a deliberate act. This cheaply blocks a
  script from flooding the endpoint to probe/reverse-engineer the detector or
  spam classifications, while leaving huge headroom for real use.
- **100 per day (sustained guard):** Caps how much a single IP can extract or
  abuse over a day even while staying under the per-minute limit. A prolific
  writer posting a handful of pieces a day never approaches it.
- **Per-IP keying** (`get_remote_address`) is a pragmatic default; a production
  system would key on the authenticated account instead.

**Evidence** — 12 rapid requests against the 10/min limit (first 10 pass, rest
are throttled):

```
request 1  -> 200      request 7  -> 200
request 2  -> 200      request 8  -> 200
request 3  -> 200      request 9  -> 200
request 4  -> 200      request 10 -> 200
request 5  -> 200      request 11 -> 429
request 6  -> 200      request 12 -> 429
```

## Audit Log

Every classification and appeal is appended as one JSON object per line to
`logs/audit.jsonl` (structured — never `print()`), readable live via `GET /log`
(newest first). A **classification** entry records `content_id`, `creator_id`,
ISO-8601 UTC `timestamp`, `attribution`, combined `confidence`, **both individual
signal scores** (`llm_score`, `stylometric_score`), the `signals` used, and
`status`. An **appeal** entry records the `content_id`, `appeal_reasoning`, the
linked original decision, and `status: "under_review"`.

Sample log (three classifications spanning all verdicts + one appeal):

## Known Limitations

**Repetition-heavy, simple-vocabulary poetry (e.g. a villanelle or nursery
rhyme) is misread as AI.** This is a direct consequence of the stylometric
signal's design: deliberate refrains and a small, repeated word set drive the
type-token ratio *down* and sentence-length variance *down* — the exact
structural fingerprint the signal treats as "AI uniformity." So a legitimate,
very human poetic form pushes Signal 2 toward a high AI score. The demanding 0.75
threshold and the LLM's semantic read mitigate this (a strong poem often reads
human to the LLM), but a plain, repetitive poem can still land in `uncertain` or
worse. The root cause is that stylometrics measures *form*, and some human forms
are intentionally uniform.

**Two further weak spots**, both tied to signal properties: (1) **very short
submissions (< ~40 words)** make the stylometric statistics meaningless — the
short-text damping pulls Signal 2 toward 0.5, so the verdict leans almost
entirely on the LLM, quietly collapsing the "two independent signals" design; and
(2) **lightly-edited AI text** defeats both signals at once — a human pass adds
the burstiness and voice that both the stylometric and LLM signals key on, so the
system under-detects it (a false negative), as seen in the "edited AI" test case
(0.30 → likely human).

## Spec Reflection

**How the spec guided me:** deciding the confidence thresholds *in `planning.md`
before writing code* — the "protect humans" stance with a demanding 0.75 AI
threshold and a wide uncertain band — turned out to be the decision that
everything else leaned on. When I later tested a formal, non-native-style human
paragraph, it scored 0.55 and landed in `uncertain` instead of being falsely
accused. That was not a lucky accident at coding time; it was a design decision
made up front and then simply implemented. Writing the three label variants
verbatim in the spec first had the same effect — the label code had a concrete
target instead of being improvised.

**How the implementation diverged:** `planning.md` assumed type-token ratio would
be a useful stylometric discriminator. When I measured the actual metrics on the
sample texts, TTR clustered at 0.86–0.90 for *every* input — it is heavily
length-confounded and near-useless on short submissions. So I diverged from the
spec: I demoted TTR to a secondary 0.3 weight, promoted sentence-length variance
(burstiness) to the primary driver, **deliberately excluded average sentence
length** as an AI indicator (long sentences signal formality, and flagging them
would manufacture false positives against formal humans), and **added short-text
damping** that was not in the original plan. The divergence was driven by
measured data contradicting an assumption in the spec.

## AI Usage

**1. Designing the Signal 1 judge prompt (M3).** I directed the AI to write the
Groq detection prompt with a reliably parseable output. It produced a JSON-mode
prompt returning `{"ai_probability", "reason"}` with `json.loads` parsing — a good
call over scraping a decimal from prose. I revised it in two ways: I added an
explicit false-positive guardrail ("formal, polished, or non-native-English
writing is still HUMAN … calling a human's work AI is the costlier mistake"),
because the raw prompt over-flagged formal text; and I kept a regex fallback in
the parser so a malformed reply degrades to a neutral 0.5 instead of crashing.

**2. Designing the stylometric scoring (M4).** I directed the AI to map the
computed metrics to a single P(AI) score per the plan. Rather than accept its
first mapping, I had it print the raw metrics on the calibration texts first —
which revealed that type-token ratio was useless at these lengths (all
0.86–0.90). I then overrode the spec: burstiness became the primary driver, TTR
was demoted, average sentence length was excluded entirely (to avoid
false-positiving formal humans), and short-text damping was added. The AI
implemented the revised design; the design change came from inspecting data.

**3. Scaffolding from the reference repo (M0).** I directed the AI to generate the
Flask skeleton by adapting the RepairSafe reference's `config`/`auditor` patterns.
It produced a working structure, but its `audit.py` used a `dict | None` type
annotation that crashes at import on this project's Python 3.9. I caught it and
had a `from __future__ import annotations` added, then smoke-tested every endpoint
before trusting the scaffold.
